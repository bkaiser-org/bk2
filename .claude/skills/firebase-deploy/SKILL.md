# Firebase Deployment

Firebase project: `bkaiser-org` (see `.firebaserc`). Hosting + Firestore + Storage + Functions are all configured in `firebase.json`.

## Prerequisite: generate the environment

`environment.ts` is generated by `set-env.js` and is **git-ignored** — never commit it (nor `set-env.js`). Before building/deploying an app:

```sh
source ./apps/<app>/.env          # load env vars (never commit .env)
ts-node ./set-env.js              # generate environment.ts (dev or prod) per app
```

`set-env.js` writes the dev or prod `environment.ts` per app. GCP Secret Manager exists for the (abandoned) App Hosting path and is currently used **only by the Cloud Functions**.

**Never** write security-sensitive config (API keys, access tokens) into a committed file.

## Deploy the app (hosting)

```sh
pnpm nx build <app> --configuration production   # build prod bundle first
firebase deploy --only hosting:<site-id>          # deploy one site
```

## Deploy Cloud Functions

**Always use the wrapper — not `firebase deploy --only functions` directly.** `pnpm run deploy:functions` runs [scripts/deploy-functions.mjs](../../../scripts/deploy-functions.mjs): build → write the buildpack pnpm pin into `dist/apps/functions/package.json` → prune prod deps → deploy.

```sh
pnpm run deploy:functions             # build + prune + deploy ALL functions
pnpm run deploy:functions fnA fnB     # same prep, deploy only these (still canary-gated)
```

Deploy **one or a few** by passing names. (Never raw `firebase deploy --only functions:<name>` — it skips the prune, leaving the dev `pnpm-lock.yaml` in `dist/apps/functions`, and the buildpack fails with `[ERR_PNPM_LOCKFILE_CONFIG_MISMATCH] … overrides configuration doesn't match the lockfile`.) The prune regenerates a prod-only lockfile matching the pruned `package.json`, which is what the buildpack installs from.

### pnpm-pin canary

The buildpack pin comes from the root [package.json](../../../package.json) `packageManager` (override with `FUNCTIONS_PNPM=<version>`). A pin the buildpack can't activate (corepack signature mismatch) or that can't read the lockfile fails the build for **all** functions. So when the pin differs from the last-verified value in [scripts/functions-pnpm-verified.json](../../../scripts/functions-pnpm-verified.json), the script first **canary-deploys one trivial function** (`getEcho` by default; `FUNCTIONS_CANARY=<name>` to change) — this exercises the exact buildpack+pnpm path with minimal blast radius (the build is shared across all functions, so any one function tests it). On success it records the new verified pin (**commit that file**) and deploys the rest; on failure it aborts before touching other functions, and you revert the pin or set `FUNCTIONS_PNPM` to the known-good value.

Functions live in `apps/functions/src/` (sub-modules: `auth`, `matrix`, `matrix-simple`, `oidc-bridge`, `replication`), built with esbuild.

### esbuild bundling gotcha: bare imports colliding with root files

`tsconfig.base.json` sets `baseUrl: "."`, so esbuild resolves **bare imports against the repo root first**. A root-level file whose name matches a package shadows the npm package — e.g. `import cors from 'cors'` resolved to the root `cors.json` (a Storage-CORS config array), so `cors(...)` threw `TypeError: … is not a function` and Firebase's codebase analysis failed for **all** functions. Fix: mark the package **`external`** in [apps/functions/project.json](../../../apps/functions/project.json) (so esbuild emits `require("cors")` instead of resolving it) and ensure it's a dependency in `apps/functions/package.json`.

### PDF generation (`generateDocument`) — Puppeteer/Chromium on Cloud Functions

Two non-obvious requirements, both already wired up — keep them:

1. **Chromium via `@sparticuz/chromium` + `puppeteer-core`** (NOT full `puppeteer`). The deploy uses pnpm, which **skips package build scripts**, so puppeteer's post-install Chrome download never runs (locally or in the buildpack) → `puppeteer.launch()` fails with "Could not find Chrome". `@sparticuz/chromium` ships a serverless Linux Chromium binary *inside* the npm package (no download needed); [browser-pool.ts](../../../apps/functions/src/pdf/browser-pool.ts) launches it via `executablePath: await chromium.executablePath()`. Both packages are `external` in `project.json`.
2. **Signed-URL IAM grant.** The function stores the PDF in Storage and returns a `getSignedUrl()` link. With the default compute runtime SA, signing needs `iam.serviceAccounts.signBlob` → grant the SA **Token Creator on itself** (one-time, per project):
   ```sh
   SA=$(gcloud functions describe generateDocument --region=europe-west6 --gen2 \
        --project=bkaiser-org --format="value(serviceConfig.serviceAccountEmail)")
   gcloud iam service-accounts add-iam-policy-binding "$SA" \
     --member="serviceAccount:$SA" \
     --role="roles/iam.serviceAccountTokenCreator" --project=bkaiser-org
   ```
   Without it, generation succeeds but the response fails with `Permission 'iam.serviceAccounts.signBlob' denied`.

## Deploy security rules

```sh
firebase deploy --only firestore:rules    # firestore.rules
firebase deploy --only storage            # storage.rules
```

CORS / CSP headers are configured under `hosting` in `firebase.json` (deployed with the app).

## Manage function secrets

```sh
firebase functions:secrets:set SECRET_NAME    # prompts for the value
```

A function that reads a new secret will fail to deploy until the secret exists; set it **before** `pnpm run deploy:functions`. Example: `firebase functions:secrets:set MAILTRAP_WEBHOOK_SECRET` before deploying the Mailtrap webhook.

## Deployment guidelines

- Keep bundles small via lazy loading (router-level component loading).
- Target first meaningful paint under 1 second.
- **Do not** use SSR, Angular hydration, or Firebase App Hosting — Ionic/Stencil are incompatible with Angular hydration. We deploy to plain Firebase Hosting.

## Common mistakes

- Running raw `firebase deploy --only functions` — skips the build/prune wrapper; use `pnpm run deploy:functions`.
- Deploying before `set-env.js` has regenerated `environment.ts` for the target environment.
- Deploying functions that read a secret before `firebase functions:secrets:set` has created it.
- Committing `environment.ts` or `set-env.js` (both git-ignored — keep it that way).
