# i18n

This project uses [Transloco](https://jsverse.github.io/transloco/) for internationalisation. The default language is **German (`de`)**; supported languages are `de`, `en`, `fr`, `es`, `it`.

---

## Key format

All translation keys start with `@`. The `TranslatePipe` and `I18nService.translate()` treat any string that does **not** start with `@` as a literal value and return it unchanged — useful when a field stores either a key or a plain string.

```
@<domain>[/<layer>].<path.to.key>
```

| Type | Example key | Resolved from |
|---|---|---|
| Main bundle | `@calevent.create` | `/assets/i18n/de.json` |
| Lazy scope | `@activity/feature.list.title` | `/assets/i18n/activity/feature/de.json` |

A key whose prefix (everything before the first `.`) contains `/` is a **lazy-loaded scope** — the file is fetched on demand the first time that prefix is encountered.

---

## Translation files

### Main bundle

`apps/scs-app/src/assets/i18n/de.json` (and `en.json`, `fr.json`, …)

Loaded once at startup via `TranslocoHttpLoader`:

```typescript
getTranslation(lang: string) {
  return this.http.get<Translation>(`/assets/i18n/${lang}.json`);
}
```

All top-level keys correspond to domain prefixes (`calevent`, `general`, `auth`, …).

### Per-module scope files

Each lib that owns translation keys keeps them in `src/i18n/de.json`:

```
libs/activity/feature/src/i18n/de.json
libs/cms/section/feature/src/i18n/de.json
…
```

The JSON root key matches the lib domain:

```json
{
  "activity": {
    "feature": {
      "list": { "title": "Aktivitäten" }
    }
  }
}
```

These files are **not bundled** — they are served as static assets and fetched lazily. The URL the loader requests is `/assets/i18n/<domain>/<layer>/de.json`, which maps to the Transloco scope key `<domain>/<layer>`.

## Scopes and the `PFX` convention

Each lib with scoped keys defines a `scope.ts` that exports a `PFX` constant:

```typescript
// libs/activity/feature/src/lib/scope.ts
export const PFX = '@activity/feature.';
```

Components import `PFX` and concatenate it with relative key paths:

```typescript
protected pfx = PFX;
// template:  {{ pfx + 'list.title' | translate | async }}
// resolves:  @activity/feature.list.title
```

---

## Template usage

```typescript
import { TranslatePipe } from '@bk2/shared-i18n';

@Component({
  imports: [AsyncPipe, TranslatePipe],
  template: `<ion-title>{{ pfx + 'list.title' | translate | async }}</ion-title>`
})
```

`TranslatePipe` delegates to `I18nService.translate()` and returns an `Observable<string>`, so `async` is required.

---

## Programmatic usage (`t()`)

For use outside templates (services, validators, utility functions):

```typescript
import { t } from '@bk2/shared-util-core';

const label = t('@calevent.create');
```

`t()` calls Transloco's synchronous `translate()` — only use it after the translation file has been loaded (i.e. not during app bootstrap).

## Resolution flow

```
template: '@activity/feature.list.title'
    │
    ▼ TranslatePipe  →  I18nService.translate()
         │
         ├─ strips '@'  →  'activity/feature.list.title'
         ├─ prefix = 'activity/feature'  (contains '/')
         │
         ▼ translocoService.load('activity/feature/de')
              │
              ▼ TranslocoHttpLoader.getTranslation('activity/feature/de')
                   GET /assets/i18n/activity/feature/de.json
                        │
                        ▼ translocoService.selectTranslate('activity/feature.list.title')
                             →  'Aktivitäten'
```

For main-bundle keys (no `/` in prefix) the `load()` step is skipped — the file is already in memory.

## Tenant overrides

After login, `I18nOverrideService.init()` queries Firestore for `I18nTenantOverride` documents scoped to the current tenant and patches the live translation map in memory:

```typescript
this.translocoService.setTranslation(
  { [`${override.module}.${override.key}`]: value },
  lang,
  { merge: true }
);
```

The `I18nTenantOverrideModel` stores one field per language (`de`, `en`, `fr`, …). Overrides take effect immediately on language change. They are managed via the CMS at `/i18n/overrides`.

## Build integration

Angular's build executor copies each lib's `src/i18n/` directory to `dist/browser/assets/i18n/<domain>/<layer>/` via the `assets` array in `apps/<app>/project.json`:

```json
{
  "glob": "*.json",
  "input": "libs/activity/feature/src/i18n",
  "output": "./assets/i18n/activity/feature"
}
```

The output path must start with `./assets/i18n/` to match the URL pattern the `TranslocoHttpLoader` requests.

### Syncing after adding a new lib

Run the sync script whenever a new lib gets an `src/i18n/de.json` file:

```sh
node scripts/sync-i18n-assets.mjs
```

The script scans all `libs/**/src/i18n/de.json` files, generates the correct asset entries, and rewrites the `assets` array in every Angular app's `project.json`. Non-Angular apps (e.g. `functions`) are skipped automatically.

---

## Adding translations to a new lib

1. Create `libs/<domain>/<layer>/src/i18n/de.json` with a root key matching the lib domain.
2. Create `libs/<domain>/<layer>/src/lib/scope.ts`:
   ```typescript
   export const PFX = '@<domain>/<layer>.';
   ```
3. Run `node scripts/sync-i18n-assets.mjs` to register the new file in all app `project.json`s.
4. Use `PFX` in components and `t('@<domain>/<layer>.<key>')` in services/utils.
