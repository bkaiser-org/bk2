Architecture Approaches

Three ways to store and identify a "schedule poll" — pick one before we spec the details

A — Pure CalEvent extension Recommended
Add state: 'proposed'|'provisional'|'definitive' to CalEventModel. Schedule dates share a seriesId (already exists). A series is a "schedule poll" when any event in it has state === 'proposed'. No new Firestore collection. Author = responsiblePersons[0]. Invited members tracked via existing invitations. The schedule table modal loads all events with the same seriesId.
B — Separate schedules collection User excluded
New Firestore collection schedules/{id} holds title, description, author, invited members, status. Each proposed CalEvent carries a scheduleId reference. Cleaner query path but extra collection, extra reads, extra service layer.
C — Schedule metadata on the first event of the series Possible
One "root" CalEvent in the series holds extra schedule fields (e.g. isScheduleRoot: boolean). Other events just carry seriesId. No new collection, but root vs. leaf distinction adds complexity to every query and makes deletion tricky.
Criterion A — Extend CalEvent B — Separate collection C — Root event
No new Firestore collection ✓ yes ✗ no ✓ yes
Reuses existing seriesId logic ✓ fully ~ parallel ✓ fully
Query "all proposed in series" ✓ filter on state+seriesId ✓ join via scheduleId ~ same as A
Model schema change 1 field added New model + service 1 boolean added
Close-poll delete operation ✓ archive all proposed in series delete schedule doc + archive events must not delete root
Aligns with user preference ✓ explicitly preferred ✗ explicitly excluded ~ acceptable
Recommendation: A
Adding state to CalEventModel is the minimal schema change. The existing seriesId, responsiblePersons, invitations collection, and subscribe/unsubscribe flow all carry over without modification. The schedule table modal is the only new component needed beyond store operations.
