# spec-pwa-caching.md

**Status:** Draft (v2 — adapted from current implementation)
**Scope:** `scs-app` PWA shell caching for Angular (standalone) + Ionic on Firebase Hosting. `test-app` and the static websites are explicitly out of scope for this iteration.
**Owner:** Bruno
**Related:** `error-monitoring.md`, `spec-sentry-integration.md`, `CHAT.md` (libs/chat/feature)

-----

## 1. Problem

On iOS, the bk2 app reloads from scratch every time Safari is returned to from the background. This is **not** a bug or a caching gap — it is iOS memory management. When Safari is backgrounded, iOS evicts the page from memory; on resume it re-requests `index.html` and re-bootstraps the app.

The forced reload cannot be prevented from web code. It can, however, be made **fast** (served from local cache instead of network), **less jarring** (Matrix sync resumes from a `since` token instead of a full initial sync, Firestore re-paints from IndexedDB), and **less disruptive** (chat scroll position, last-read marker, last-visited section, and accordion expand state are restored from `localStorage` on bootstrap). This spec covers all three.

## 2. Decision (as a delta from today)

- **Add** `@angular/service-worker` (ngsw) to `scs-app` and cache the app shell + static assets. The existing `firebase-messaging-sw.js` (FCM background push) stays as-is and registers under a separate scope (§5).
- **Add** Firebase Hosting cache headers for the ngsw control files. Hashed asset caching is already in place (§6).
- **Enable Firestore IndexedDB persistence on all browsers, including Safari**, using `persistentSingleTabManager` and a memory-cache fallback. The historical Safari carve-out is removed — implemented in [libs/shared/config/src/lib/firestore.ts](libs/shared/config/src/lib/firestore.ts), see §7 for the research that backed the removal.
- **Do not add an SwUpdate auto-reload.** Rely on iOS's natural reload-on-resume plus the existing [VersionCheckService](libs/shared/util-angular/src/lib/version-check.service.ts) (§9).
- **Rewrite §8** around Matrix (`matrix-js-sdk`), not Firestore — chat is not Firestore-backed (see [libs/chat/feature/src/lib/CHAT.md](libs/chat/feature/src/lib/CHAT.md)).
- **Add UI state persistence (§11)** for chat scroll, last-read marker, last-visited section, and accordion state. Persisted to `localStorage` via a new `persistedSignal()` helper in `@bk2/shared-util-angular`. Composable with the existing visibility hook in `app.store.ts`.
- No `dataGroups` in `ngsw-config.json`. Firestore and Matrix each own their own cache.

## 4. Registration

Add to [apps/scs-app/src/app/app.config.ts](apps/scs-app/src/app/app.config.ts), inside the existing `providers` array:

```ts
import { provideServiceWorker } from '@angular/service-worker';

provideServiceWorker('ngsw-worker.js', {
  enabled: !isDevMode() && typeof window !== 'undefined',
  registrationStrategy: 'registerWhenStable:30000',
  scope: '/',
});
```

`isDevMode()` is already imported in the file. `registerWhenStable:30000` defers registration until the app is stable (or 30 s elapses), keeping it off the critical path with Matrix init (10 s deferred) and Analytics.

**SSR guard:** the `typeof window !== 'undefined'` predicate keeps the registrar's lifecycle code off the server render path. `provideServiceWorker` itself still gets called during SSR — that is fine; with `enabled: false` it provides `SwUpdate`/`SwPush` as no-op shims so consuming code can inject them without crashing. We deliberately do not wrap this in `APP_BOOTSTRAP_LISTENER` (the pattern used elsewhere): `provideServiceWorker` returns `EnvironmentProviders` and must live in the providers array, not inside a bootstrap listener.

**Also add a one-time cleanup of the stale root-scope FCM SW registration** (§5.1 migration note) directly above the registration in the same commit:

```ts
if (typeof navigator !== 'undefined' && 'serviceWorker' in navigator) {
  navigator.serviceWorker.getRegistrations().then(regs => {
    regs
      .filter(r => r.active?.scriptURL.endsWith('/firebase-messaging-sw.js') && r.scope.endsWith('/'))
      .forEach(r => r.unregister());
  });
}
```

## 5. ngsw-config.json + FCM SW coexistence

Create `apps/scs-app/ngsw-config.json`:

```json
{
  "$schema": "./node_modules/@angular/service-worker/config/schema.json",
  "index": "/index.html",
  "assetGroups": [
    {
      "name": "app-shell",
      "installMode": "prefetch",
      "updateMode": "prefetch",
      "resources": {
        "files": [
          "/index.html",
          "/manifest.webmanifest",
          "/favicon.ico",
          "/*.css",
          "/*.js"
        ]
      }
    },
    {
      "name": "assets",
      "installMode": "lazy",
      "updateMode": "prefetch",
      "resources": {
        "files": [
          "/assets/**",
          "/*.(png|jpg|jpeg|webp|avif|gif|ico|otf|ttf|woff|woff2)"
        ]
      }
    }
  ],
  "navigationUrls": [
    "/**",
    "!/web/**",
    "!/**/*.*"
  ]
}
```

Register the config in [apps/scs-app/project.json](apps/scs-app/project.json) `build.options`:

```jsonc
"serviceWorker": "apps/scs-app/ngsw-config.json"
```

**Notes on the glob choices:**
- **No `/svg/**` glob, no `*.svg`.** All icons in bk2 use `<ion-icon src="{{ name | svgIcon }}" />` ([libs/shared/pipes/src/lib/svg-icon.pipe.ts](libs/shared/pipes/src/lib/svg-icon.pipe.ts)), where `svgIcon` returns `${imgixBaseUrl}/logo/icons/${name}.svg` — i.e. icons are served from imgix (cross-origin), NOT from the local `/svg/` directory. ngsw cannot cache cross-origin URLs unless they're declared as `dataGroups` with an explicit origin, which we are deliberately not doing. See §10.7 for the related bundle-size cleanup of the dead `ionicons/dist/ionicons/svg → ./svg` asset copy in [project.json:658-661](apps/scs-app/project.json).
- `/web/**` MUST be excluded from `navigationUrls` — the scs-website static bundle copied into `/web/` ([project.json:60-64](apps/scs-app/project.json)) has its own routing (`/web/news`, `/web/bootshaus`, …) and must not be rewritten to `/index.html`.

### 5.1 FCM SW coexistence — the actual conflict and the fix

A browser allows multiple service workers per origin, but for any URL there is exactly **one** controlling SW, determined by **scope**. ngsw needs scope `/`. The challenge: bk2 today already has a SW claiming `/`.

**Current state — two registrations of `firebase-messaging-sw.js` exist, one of them at root:**

1. Eager manual registration in [apps/scs-app/src/main.ts:43-51](apps/scs-app/src/main.ts):
   ```js
   navigator.serviceWorker.register('/firebase-messaging-sw.js')
   // no scope option → defaults to '/' (the script's directory). Claims ROOT scope.
   ```
2. Implicit registration inside the Firebase Messaging SDK when [fcm.service.ts:107](libs/shared/data-access/src/lib/fcm.service.ts) calls `getToken(this.messaging, { vapidKey })` without passing a `serviceWorkerRegistration` — the SDK auto-registers `/firebase-messaging-sw.js` at scope `/firebase-cloud-messaging-push-scope/` on first call.

The root-scope registration blocks ngsw. Push events themselves don't care about scope (the browser dispatches them to whatever SW is registered for the FCM token), but ngsw's ability to control navigation absolutely does.

**Recommended fix — Option A: drop the manual registration. _Applied_** in [apps/scs-app/src/main.ts](apps/scs-app/src/main.ts) — the eager `navigator.serviceWorker.register('/firebase-messaging-sw.js')` block has been removed. The FCM SDK auto-registers `firebase-messaging-sw.js` at `/firebase-cloud-messaging-push-scope/` on the first `getToken()` call inside `FcmService.registerWebAndSave` ([libs/shared/data-access/src/lib/fcm.service.ts:107](libs/shared/data-access/src/lib/fcm.service.ts)). ngsw will be free to own `/` once it ships.

The (tiny) downside: until the user grants notification permission and `getToken` runs once, the FCM SW isn't installed — so a hypothetical background push received in the window between page load and permission grant would not fire `onBackgroundMessage`. This is moot in practice: pushes are addressed to tokens, and there is no token until `getToken` runs.

**Migration concern for existing installs.** Users who already loaded a previous build have a persisted `firebase-messaging-sw.js` registration at root scope `/` in their browser. That registration is not removed by simply deleting `main.ts:register()` — it persists across deploys until the SW is explicitly `unregister()`ed or the user clears site data. The new build (sans manual registration) won't add a new root-scope SW, but the old one stays.

This becomes a real conflict when ngsw is later added: ngsw will try to register at `/` and either fail or unregister the existing FCM SW. To avoid surprises, the ngsw-rollout commit should also add a one-time cleanup in `main.ts` before `provideServiceWorker` does its work:

```ts
if (typeof navigator !== 'undefined' && 'serviceWorker' in navigator) {
  navigator.serviceWorker.getRegistrations().then(regs => {
    regs
      .filter(r => r.active?.scriptURL.endsWith('/firebase-messaging-sw.js') && r.scope.endsWith('/'))
      .forEach(r => r.unregister());
  });
}
```

After unregister, the next `FcmService.registerWebAndSave` call will auto-register FCM under the correct push-scope.

**Alternative — Option B: keep an eager registration, but at a non-root scope:**
```js
// main.ts
navigator.serviceWorker
  .register('/firebase-messaging-sw.js', { scope: '/firebase-cloud-messaging-push-scope/' })
```
…and in [fcm.service.ts](libs/shared/data-access/src/lib/fcm.service.ts) pass it to `getToken`:
```ts
const swReg = await navigator.serviceWorker.getRegistration('/firebase-cloud-messaging-push-scope/');
const token = await getToken(this.messaging, { vapidKey, serviceWorkerRegistration: swReg });
```
More code, FCM SW is installed eagerly. Pick this only if a concrete reason emerges to install the FCM SW before the first `getToken` call.

**Verification after the change:** DevTools → Application → Service Workers shows two workers with distinct scopes — `ngsw-worker.js` at `/`, `firebase-messaging-sw.js` at `/firebase-cloud-messaging-push-scope/`. A test background push still fires `onBackgroundMessage`.

### 5.2 Deliberately excluded — no `dataGroups`

- Firestore runs over `firestore.googleapis.com` (cross-origin; ngsw would not intercept it regardless) and the Firebase SDK owns its own cache (§7).
- Matrix runs over `matrix.bkchat.etke.host` (cross-origin) and matrix-js-sdk owns its IndexedDB store (§8).
- Imgix (`*.imgix.net`) serves images and SVG icons cross-origin. It already sets long `Cache-Control` headers, so the browser HTTP cache covers reload-warm fetches. Adding a `dataGroup` here would only help true-offline use; out of scope.
- Auth — never cache `identitytoolkit.googleapis.com` or `securetoken.googleapis.com`.

## 6. firebase.json header delta (scs-app site only)

Principle: hashed assets are immutable; control files revalidate.

The existing [firebase.json](firebase.json) for `hosting[0]` (`scs-app-54aef`) already covers JS/CSS immutability and HTML revalidation. Add the following entries to the `headers` array for that site **only** (not test-app or the websites):

```jsonc
{
  "source": "/@(ngsw-worker.js|ngsw.json|safety-worker.js|firebase-messaging-sw.js|firebase-config.js|manifest.webmanifest)",
  "headers": [{ "key": "Cache-Control", "value": "no-cache" }]
},
{
  "source": "**/*.@(jpg|jpeg|png|gif|svg|webp|avif|woff|woff2|ttf|otf|ico)",
  "headers": [{ "key": "Cache-Control", "value": "public, max-age=31536000, immutable" }]
}
```

**Caveats:**
- `no-cache` means "store but revalidate before use," not "do not store." Control files still benefit from 304s while staying fresh — this is what prevents an iOS device from pinning a stale `ngsw-worker.js`.
- The `immutable` rule on images/fonts is only safe because Angular content-hashes asset filenames in production (`outputHashing: all`). Anything copied verbatim into `dist/apps/scs-app/browser` and referenced by a stable name (e.g. `favicon.ico`, `manifest.webmanifest`) must NOT be matched by the immutable rule — the source list above intentionally excludes them.
- Do NOT add these rules to `hosting[1]` (test-app) or the website hosting blocks in this iteration.

## 7. Firestore persistence — enabled everywhere with graceful fallback

**Status:** Implemented in [libs/shared/config/src/lib/firestore.ts](libs/shared/config/src/lib/firestore.ts).

The historical Safari carve-out (no IndexedDB persistence on Safari) has been **removed**. Research against the firebase-js-sdk issue tracker (see Evidence below) confirmed that the specific failure mode the carve-out protected against — IndexedDB refuse-to-open / corruption error loops on iOS 18 Safari — was [#9056](https://github.com/firebase/firebase-js-sdk/issues/9056), fixed by [PR #9162](https://github.com/firebase/firebase-js-sdk/pull/9162) and shipped in firebase-js-sdk **v11.10** (Jul 2025). bk2 is on `^12.8.0`, well past the fix.

### 7.1 What changed

`initializeFirestore` now uses:

| Layer | Setting | Rationale |
|---|---|---|
| Transport (Safari) | `experimentalForceLongPolling: true` | Unchanged. The WebChannel/WebSocket reliability issue on Safari/iOS is independent of the IndexedDB issue and still needs the workaround. |
| Transport (non-Safari) | `experimentalAutoDetectLongPolling: true` | Unchanged. |
| Cache (all browsers) | `persistentLocalCache({ tabManager: persistentSingleTabManager({}) })` | Restores instant `metadata.fromCache === true` paints on reload. Single-tab manager sidesteps the open multi-tab leader-election issues ([#6511](https://github.com/firebase/firebase-js-sdk/issues/6511), [#8314](https://github.com/firebase/firebase-js-sdk/issues/8314), [#6806](https://github.com/firebase/firebase-js-sdk/issues/6806)) which surface most often on Safari. |
| Eviction hint | `navigator.storage.persist()` fire-and-forget | Harmless in a browser tab. Materially helps when bk2 is installed as a Home Screen PWA — WebKit's [Aug 2023 storage policy](https://webkit.org/blog/14403/updates-to-storage-policy/) excludes persisted origins from LRU eviction and gives Home Screen Web Apps browser-tier quotas. |
| Failure path | `try { persistent } catch { memoryLocalCache() }` | Covers the still-open [#8860](https://github.com/firebase/firebase-js-sdk/issues/8860) (Safari IndexedDB read-error loop, low-volume, not reproduced in 12.x). On failure the app keeps working — just without offline cache. |

Multi-tab manager remains an option for a future desktop-multi-tab use case; gate it as `isSafariBrowser ? persistentSingleTabManager({}) : persistentMultipleTabManager()` when needed.

### 7.2 Why this is safe to roll out

- **The historical bug is fixed.** Two relevant Safari fixes shipped in 2025 — v11.8.0 (May 2025, client-side indexing cache) and v11.10 (#9162). bk2's `^12.8.0` includes both.
- **Long polling and persistent cache are orthogonal.** No documented interaction bug.
- **Single-tab manager avoids the active risk surface.** The unresolved Safari issues are almost all multi-tab leader-election scenarios. Single-tab eliminates that class entirely.
- **Graceful degradation if IndexedDB is unavailable.** ITP storage eviction (or an unforeseen init failure) falls back to in-memory cache — the app continues to work, just paying a cold-fetch on each reload.

### 7.3 What this gets us

- Re-attached `onSnapshot` listeners on iOS resume now serve cached data from IndexedDB immediately (`metadata.fromCache === true`), then reconcile deltas. The thread paints instantly instead of blank-then-spinner.
- Queued writes that were mid-send when iOS killed the page are persisted in IndexedDB and flush on reconnect.
- Reduced read cost on every reload — only the delta sync hits Firestore.

### 7.4 iOS Safari caveat (unchanged)

Even with `navigator.storage.persist()` granted, Safari may still evict IndexedDB under severe storage pressure or in tabs (not Home Screen installs) that go long periods without user interaction. Treat the Firestore cache as a **latency optimization, not a guarantee** — the listener always reconciles against the server, so an evicted cache degrades to a normal cold fetch rather than a broken state.

### 7.5 Evidence

The carve-out removal was validated against:

- firebase-js-sdk release notes — v11.8.0 (Safari/WebKit cache fix, May 2025) and v11.10 (#9162, Jul 2025).
- Open and closed issues: [#9056 (fixed)](https://github.com/firebase/firebase-js-sdk/issues/9056), [#8593 (Clear Site Data)](https://github.com/firebase/firebase-js-sdk/issues/8593), [#8860 (open, low-volume — covered by memory fallback)](https://github.com/firebase/firebase-js-sdk/issues/8860).
- WebKit's [Updates to Storage Policy (Aug 2023)](https://webkit.org/blog/14403/updates-to-storage-policy/) — LRU-under-pressure model, `persist()` excludes from eviction, Home Screen Web Apps favored.
- [Firebase docs: Access data offline](https://firebase.google.com/docs/firestore/manage-data/enable-offline) — no current Safari-specific caveat documented.

## 8. Chat module (Matrix) — reload behaviour

Per [libs/chat/feature/src/lib/CHAT.md](libs/chat/feature/src/lib/CHAT.md), the chat is Matrix-backed via `matrix-js-sdk` talking to `matrix.bkchat.etke.host`. **The service worker does not touch any of this** — Matrix traffic is cross-origin and not in the ngsw scope.

### 8.1 What's already correct

- Init is **already** gated on auth and deferred: [apps/scs-app/src/app/app.config.ts:91-110](apps/scs-app/src/app/app.config.ts) waits for `AppStore.currentUser`, skips `/public/*` routes, then `setTimeout(…, 10000)` before dynamic-importing `MatrixInitializationService`. This avoids the "permission-denied on cold reload" failure mode and keeps `matrix-js-sdk` (~700 KB) out of the initial bundle. No change needed.
- The 10 s defer also means matrix-js-sdk does not compete with `ngsw-worker.js` registration (which fires up to 30 s after stability) for the critical-path bandwidth.

### 8.2 Persist the Matrix sync token

Today, every iOS resume triggers a full initial `/sync` against the homeserver — slow and bandwidth-heavy. matrix-js-sdk supports an `IndexedDBStore` (in-browser) that persists rooms, members, and the `since` token between page loads. On reload, the SDK resumes from the stored `since` and only fetches deltas.

Wire it in `MatrixChatService.initialize()` ([libs/chat/data-access/src/lib/matrix-chat.service.ts](libs/chat/data-access/src/lib/matrix-chat.service.ts)):

```ts
import { createClient, IndexedDBStore } from 'matrix-js-sdk';

const store = new IndexedDBStore({
  indexedDB: window.indexedDB,
  localStorage: window.localStorage,
  dbName: 'bk2-matrix',
});
await store.startup();

const client = createClient({
  baseUrl: token.homeserverUrl,
  accessToken: token.accessToken,
  userId: token.userId,
  deviceId: token.deviceId,
  store,
});
await client.startClient({ initialSyncLimit: 20 });
```

Effect: on iOS reload, the chat list paints near-instantly from the store, and `/sync` runs incrementally. `initialSyncLimit: 20` caps the per-room timeline so cold sync is bounded.

### 8.3 Treat presence/typing as ephemeral

Presence and typing indicators must not be persisted or restored — let them re-establish on reconnect. This already matches current behaviour; no change.

### 8.4 Crypto store (optional)

If/when E2EE is enabled, matrix-js-sdk also exposes an `IndexedDBCryptoStore` for device keys. Out of scope here — file as a follow-up if E2EE is enabled.

### 8.5 iOS Safari caveat

Same ITP eviction caveat as Firestore (§7). matrix-js-sdk handles missing store gracefully — falls back to a full initial sync.

## 9. Version updates — explicit non-decision

The spec does **not** add an `SwUpdate.versionUpdates` auto-reload subscription.

Rationale:
- iOS already forces a reload on resume, which is precisely when ngsw activates the new shell. The user-visible behaviour is "reload picks up the new version" without any extra code.
- The existing [VersionCheckService](libs/shared/util-angular/src/lib/version-check.service.ts) already streams the `app-version` Firestore doc and prompts the user with an alert when a min/latest version mismatch is detected, then calls `window.location.reload()` on confirm. This is the admin-controlled forced-update path and stays as the source of truth.
- Adding a second reload path from `SwUpdate` would risk reloading mid-form-input and would duplicate (or race) the admin-controlled flow.

If a future need emerges (e.g. "force a reload on every new build, even without admin push"), revisit in a follow-up — the swap-in is small.

## 10. Bundle-size recommendations (separate from caching, but related)

The smaller the initial bundle, the cheaper every iOS forced reload is — even with ngsw caching the bytes, the parse / hydrate cost matters. These are diagnosed, not yet validated; treat as a backlog of investigations.

1. **Audit the `allowedCommonJsDependencies` list** ([apps/scs-app/project.json:677-693](apps/scs-app/project.json)). Anything listed there is a CommonJS dep, which Angular cannot tree-shake well. Candidates to defer behind a dynamic import:
   - `jszip`, `file-saver` — used for export? Likely a single feature; dynamic-import at the call site.
   - `crypto-js` — used where? If in one feature, defer.
   - `fix-webm-duration` — recording-only; defer to the recorder component.
   - `dayjs/plugin/calendar`, `dayjs/plugin/relativeTime` — load only on routes that format relative times.
2. **Lazy-load Transloco translation JSONs.** Today every `*.json` under `libs/**/i18n` is copied via separate asset entries in [project.json:67-655](apps/scs-app/project.json). Transloco can scope these — confirm the runtime fetches only the active language. Five languages × N libs = a lot of HTTP if not scoped per route.
3. **Defer `@angular/fire`-style analytics** — already done via `AnalyticsLoaderService` ([apps/scs-app/src/app/app.config.ts:114-125](apps/scs-app/src/app/app.config.ts)). Verify the production bundle doesn't ship `firebase/analytics` eagerly (look for it in the chunk graph after `pnpm nx build scs-app --configuration production --stats-json`, then inspect with `webpack-bundle-analyzer` or `source-map-explorer`).
4. **`matrix-js-sdk`** — already dynamic-imported and deferred 10 s. Confirm the chunk it lands in does not pull in `crypto-js` or other shared CJS deps at static-analysis time.
5. **Splitting the `/web/**` static bundle.** The scs-website assets are copied into the scs-app build ([project.json:60-64](apps/scs-app/project.json)). They add disk + cache surface but should NOT be in the ngsw asset group — `navigationUrls` already excludes `/web/**` in §5. Verify the ngsw glob doesn't accidentally swallow `/web/**/*.js` etc.
6. **Set a `budgets.initial` warning at the size you actually want.** Today: 8 MB warning / 10 MB error ([project.json:701-705](apps/scs-app/project.json)). For an iOS PWA hitting first paint <1 s (per CLAUDE.md), aim for <500 KB initial gzipped. Tighten budgets to flag regressions early.
7. **Remove the dead `ionicons/dist/ionicons/svg → ./svg` asset copy.** [project.json:658-661](apps/scs-app/project.json) copies every ionicons SVG into the build output under `./svg`. These files are never referenced at runtime — bk2 uses `<ion-icon src="{{ name | svgIcon }}" />` exclusively, and `svgIcon` returns an imgix URL (`${imgixBaseUrl}/logo/icons/${name}.svg`), not a `/svg/...` local URL. Removing the asset entry cuts the deploy bundle and the ngsw asset surface for free. Verify with `grep -rn "ion-icon" libs apps | grep 'name='` — expect zero hits in templates.
8. **Icon caching** — since icons come from imgix (cross-origin), they live in the browser HTTP cache governed by imgix's response headers, not in the ngsw asset group. First cold load is one HTTP request per distinct icon. If this becomes a measured problem, the answer is an imgix `Cache-Control: immutable, max-age=31536000` confirmation and/or icon sprite consolidation — not a ngsw `dataGroup`.

A focused follow-up spec ("spec-bundle-size.md") could turn each of these into a measured win with before/after numbers.

## 11. UI state persistence (in scope)

The service worker makes reloads **fast** (§4–§6); §7 makes Firestore re-paint instantly; §8 makes Matrix sync resume from a `since` token. The remaining gap is **in-memory UI state**: when iOS hard-reloads, the user lands at the right URL but loses scroll position, chat draft (everywhere except where §11.2 already exists), expansion state, and current selection. This section makes that gap closeable.

### 11.1 What's already in place

- **Per-room chat draft** — [matrix-message-input.ts:481](libs/chat/ui/src/lib/matrix-message-input.ts) already persists the message textarea per room key in `localStorage` via `onTyping`. Pattern to follow for the rest.
- **Visibility hook** — [app.store.ts:389-410](libs/shared/feature/src/lib/app.store.ts) already runs on `visibilitychange` and Capacitor `appStateChange` to end/restart the session. This is the natural place to **flush** any pending writes before iOS evicts the page.
- **Active route** — does NOT need persistence. The browser preserves the URL across the iOS reload; Angular Router navigates straight to it on bootstrap. (Hash and query params survive the same way.)

### 11.2 Scope of new persistence

Persist (small / high-value):

| Slice | Key shape | Where written | Where read |
|---|---|---|---|
| Chat scroll position per room | `bk2:${tenantId}:${uid}:chat:scroll:${roomId}` → `{ topMessageId, offsetPx }` | Debounced on `IonContent` scroll event in the chat view | On `messagesResource` first paint per room |
| Chat last-read marker per room | `bk2:${tenantId}:${uid}:chat:lastRead:${roomId}` → `eventId` | On message-visible event in the chat view | On room entry to compute unread badge |
| Last-visited section per app surface | `bk2:${tenantId}:${uid}:nav:lastSection` → `string` | Router event listener | `APP_INITIALIZER` on cold boot |
| Section/list expand-collapse state per page | `bk2:${tenantId}:${uid}:ui:expand:${pageKey}` → `string[]` of expanded section keys | On accordion toggle | On section list first paint |

Out of scope (explicit):

- **Unsaved form input** — too form-specific; `ngx-vest-forms` plus template-driven forms make a generic serializer brittle. If a specific form needs draft persistence later, follow the chat-draft pattern at the form-component level.
- **Modal / overlay state** — modals are inherently transient; reload should always start with overlays closed.
- **Anything ephemeral that Matrix or Firestore would re-emit on reconnect** (presence, typing indicators, live counts).

### 11.3 Storage choice — localStorage

For everything in §11.2, `localStorage` is the right primitive:

- Sync API → no orchestration around async reads inside `APP_INITIALIZER`.
- Quota (5–10 MB) is far above what these slices need.
- Already used by chat drafts and Matrix tokens — same eviction profile across the app.
- IndexedDB would only be justified if a slice grew past a few hundred KB or required structured queries (neither applies here).

iOS Safari evicts localStorage under the same WebKit storage-policy rules as IndexedDB (§7.4). Same mitigations apply: `navigator.storage.persist()` already requested in `FIRESTORE` factory ([libs/shared/config/src/lib/firestore.ts](libs/shared/config/src/lib/firestore.ts)) covers localStorage too — the persist grant is per-origin, not per-storage-type.

### 11.4 Helper — `persistedSignal()`

Build one shared signal helper in `@bk2/shared-util-angular` so every slice has the same write/read semantics:

```ts
// libs/shared/util-angular/src/lib/persisted-signal.ts
import { effect, signal, WritableSignal } from '@angular/core';

interface Options {
  debounceMs?: number;   // default 250
  ttlMs?: number;        // default 30 days; reads return initialValue if older
}

interface Envelope<T> { v: T; t: number }

export function persistedSignal<T>(
  key: string,
  initialValue: T,
  opts: Options = {},
): WritableSignal<T> {
  const { debounceMs = 250, ttlMs = 30 * 24 * 60 * 60 * 1000 } = opts;

  let restored = initialValue;
  if (typeof localStorage !== 'undefined') {
    try {
      const raw = localStorage.getItem(key);
      if (raw) {
        const env = JSON.parse(raw) as Envelope<T>;
        if (Date.now() - env.t < ttlMs) restored = env.v;
      }
    } catch { /* corrupted entry — ignore */ }
  }

  const sig = signal<T>(restored);
  let pending: ReturnType<typeof setTimeout> | undefined;

  effect(() => {
    const value = sig();
    if (typeof localStorage === 'undefined') return;
    if (pending) clearTimeout(pending);
    pending = setTimeout(() => {
      try {
        localStorage.setItem(key, JSON.stringify({ v: value, t: Date.now() } satisfies Envelope<T>));
      } catch { /* quota exceeded — drop */ }
      pending = undefined;
    }, debounceMs);
  });

  // Flush on backgrounding so iOS doesn't evict mid-debounce.
  if (typeof document !== 'undefined') {
    document.addEventListener('visibilitychange', () => {
      if (document.visibilityState === 'hidden' && pending) {
        clearTimeout(pending);
        pending = undefined;
        try {
          localStorage.setItem(key, JSON.stringify({ v: sig(), t: Date.now() } satisfies Envelope<T>));
        } catch { /* ignore */ }
      }
    });
  }

  return sig;
}
```

Notes:
- The `Envelope<T>` wrapper enables TTL invalidation and a future migration path (add `version` field if a slice's schema changes).
- The `visibilitychange` flush composes with the existing handler in [app.store.ts:389](libs/shared/feature/src/lib/app.store.ts) — both fire on the same event without interfering.
- Read happens once at signal creation, so consumers must construct the signal in a stable place (component constructor or store `withProps`). Don't re-create it inside `computed`.

### 11.5 Migration / cleanup

- Reads tolerate missing or malformed entries by returning `initialValue` — no migration code needed for the first rollout.
- Tenant-switching: store keys are tenant- and user-scoped (`bk2:${tenantId}:${uid}:…`) so a tenant switch naturally reads a separate slice. No explicit purge needed.
- Logout: optional follow-up to walk `localStorage` and remove `bk2:${tenantId}:${uid}:*` keys for the departing user. Not required for correctness — the next login simply reads a different namespace.

### 11.6 Rollout order

1. Land `persistedSignal()` in `@bk2/shared-util-angular` with a unit test for round-trip, TTL expiry, and the `visibilitychange` flush.
2. Convert the existing chat draft persistence in [matrix-message-input.ts](libs/chat/ui/src/lib/matrix-message-input.ts) to use `persistedSignal()` — proves the pattern against an existing use case.
3. Add chat scroll position + last-read marker.
4. Add last-visited section.
5. Add expand-collapse state for section lists.

Each step is independently shippable. Steps 3-5 can land after the ngsw rollout — they have no dependency on the service worker.
