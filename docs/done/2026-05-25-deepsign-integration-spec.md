# DeepSign Integration — Specification

**Version:** 1.0
**Status:** Draft
**Target API:** DeepSign API v1 — `https://api.sign.deepbox.swiss/api/v1`
**Reference:** https://apidocs.deepcloud.swiss/deepsign-api-docs/

---

## 1. Goal & Scope

Integrate DeepSign (DeepCloud) as the e-signature provider for our platform. PDF documents that the user wants signed are either:

- **uploaded by the user** (any PDF), or
- **generated by our CF PDF-generator** (templated PDFs from our system).

In both cases, the PDF carries embedded **Text Field Patterns** that DeepSign scans on upload to derive signees and signature positions. As a result, our integration only needs the **single happy-path call** `POST /api/v1/documents/file` (with `scanPredefined=true`) followed by `PUT /api/v1/documents/{documentId}/start`, plus **webhook callbacks** for status updates.

A new **EsignListPage** (and corresponding **EsignViewModal**) gives the user visibility and control over signing processes. Every signing call against DeepSign creates exactly **one record** in the EsignList.

Out of scope for v1: hash signing, company seals, ad-hoc signee/observer management via API, manual placement of signature fields, batch upload, attachments.

1. User initiates upload + send from EsignListPage context menu.

## 3. Visual Signature — Text Field Patterns

DeepSign supports embedded **Text Field Patterns** (visible text tokens in the PDF) which it parses on upload when `scanPredefined=true` is set. Our PDF-generator and any user-uploaded PDF must include one pattern per signee at the desired signature location.

### 3.1 Pattern Syntax

A Text Field Pattern is a text token rendered into the PDF at the position where the signature box should appear. The exact rendering rules from the DeepSign docs apply; the placeholder text itself is removed and replaced by the signature widget on processing.

Recommended pattern fields:

| Field | Required | Description |
|---|---|---|
| `name` | yes | Display name of the signee (used in invitation emails). |
| `email` | yes | Email address — DeepSign sends the invitation to this address. |
| `signOrder` | no | Order index when sequential signing is required (0-based). Omit for parallel signing. |
| `signatureType` | no | `signature` (default) or `seal`. |

The bounding box of the rendered pattern text determines the position, page, and size of the resulting signature field. Use the format documented under DeepSign → *Visual Signature → Text Field Pattern*.

### 3.2 CF PDF-Generator Integration

The PDF-generator must expose a template variable `signatureBlocks: SignatureBlock[]` where each block carries `{ name, email, signOrder?, page, x, y, width, height }`. The generator renders the corresponding pattern text inside the given bounding box.

For user-uploaded PDFs, the user is responsible for embedding the patterns themselves (e.g. by using a Word template that we provide). The upload flow optionally calls `POST /documents/file/scan-predefined` first as a **dry-run validation** to confirm at least one signee is detected before the actual upload.

## 4. Cloud Functions

All Cloud Functions live under `functions/src/esign/`. The DeepSign access token is fetched via OAuth2 (Keycloak / `deepcloud.swiss/auth/realms/sso`) using the service-account credentials stored as `DEEPSIGN_CLIENT_ID`, `DEEPSIGN_CLIENT_SECRET`, `DEEPSIGN_SERVICE_USERNAME`, `DEEPSIGN_SERVICE_PASSWORD` in Secret Manager. A shared helper `getDeepSignAccessToken()` caches the token in memory until 60 s before expiry.

### 4.1 `esignScanPredefined` (callable, optional dry-run)

**Purpose:** Validate a PDF contains valid Text Field Patterns before committing to upload.

**Input:**
```ts
{
  storagePath: string;   // gs://… path of the PDF in our Storage bucket
}
```

**Output:**
```ts
{
  signatureFields: Array<{
    signFieldName: string;
    email: string;
    signOrder: number;
    signatureType: 'signature' | 'seal';
    autographPosition: { pageNumber: number; x: number; y: number; width: number; height: number };
  }>;
  observers: Array<{ email: string; isAdmin: boolean }>;
  signatureMode: string;
  jurisdiction: string;
}
```

**Behaviour:** Downloads the file from Storage, calls `POST /api/v1/documents/file/scan-predefined` (multipart), and returns the parsed result. Does **not** create an esignList record.

### 4.2 `esignSendDocument` (callable, primary entry point)

**Purpose:** Upload a PDF to DeepSign and start the signing process. Creates exactly one record in `/esignList`.

**Input:**
```ts
{
  storagePath: string;          // gs://… path of the PDF
  documentName?: string;        // defaults to filename
  initiatorAliasName: string;   // display name shown in invitation emails
  comment?: string;             // message shown to signees
  signatureMode?: 'timestamp' | 'advanced' | 'qualified';   // default 'timestamp'
  jurisdiction?: 'zertes' | 'eidas';                         // default 'zertes'
  sendMail?: 'all' | 'others' | 'none';                      // default 'all'
  source: 'user-upload' | 'cf-generated';
  sourceRef?: string;           // optional ref to the originating record (e.g. invoiceId)
}
```

**Steps:**

1. Verify caller is authenticated; resolve `userId`, `tenantId`.
2. Pre-create an `esignList` doc with status `uploading`, generate `esignId` (Firestore auto-ID).
3. Download PDF from Storage to memory (size ≤ 40 MB — enforced).
4. Acquire DeepSign access token.
5. `POST /api/v1/documents/file` — multipart, with `data`:
   ```json
   {
     "initiatorAliasName": "<initiatorAliasName>",
     "sendMail": "<sendMail>",
     "comment": "<comment>",
     "signatureMode": "<signatureMode>",
     "jurisdiction": "<jurisdiction>",
     "scanPredefined": true,
     "callbacks": {
       "documentStatusUrl": "https://<region>-<project>.cloudfunctions.net/esignWebhook?esignId=<esignId>&token=<hmac>",
       "signeeStatusUrl":   "https://<region>-<project>.cloudfunctions.net/esignWebhook?esignId=<esignId>&token=<hmac>"
     }
   }
   ```
   The `<hmac>` is `HMAC-SHA256(esignId, WEBHOOK_SECRET)` — used to authenticate the webhook (see §5).
6. Persist returned `documentId`, `documentStatus`, `signees`, `signeesOrdered`, `creationTime` to the esignList record. Status moves to `draft`.
7. `PUT /api/v1/documents/{documentId}/start` — start the signing process. Status moves to `in-progress` (also updated locally).
8. Return `{ esignId, documentId, signees }`.

**Failure handling:** If steps 5–7 fail, esignList record is updated to status `error` with `errorMessage`. The PDF in Storage is retained for retry.

### 4.3 `esignGetDocumentDetails` (callable)

**Purpose:** Fetch the latest `documentUrl` (short-lived) for preview/download from EsignViewModal, plus the up-to-date signee list with `viewedTime` / `signedTime`.

**Input:** `{ esignId: string }`

**Output:** the full `Document Details` payload from `GET /api/v1/documents/{documentId}` plus a `previewUrl` (re-fetched documentUrl, valid ≈ 1 min).

**Note:** DeepSign rate-limits **polling** GETs to once per 15 min, but this function is invoked on demand by the user opening the modal — well within limits. We do not poll in a loop.

### 4.4 `esignDelete` (callable)

**Purpose:** Delete a signing process completely — from DeepSign and from our esignList. This is the single destructive action exposed in the UI (no separate "Withdraw"). Used to correct a frozen document: the user deletes the process and re-uploads.

**Input:** `{ esignId: string; reason?: string }`

**Behaviour by current `documentStatus`:**

| Current status | Action |
|---|---|
| `uploading` / `error` | Delete the esignList record only; remove the staging PDF from Storage. |
| `draft` | `DELETE /api/v1/documents/{documentId}` → delete esignList record. |
| `in-progress` | `PUT /api/v1/documents/{documentId}/withdraw` → poll `GET` once to confirm `withdrawn`, then `DELETE /api/v1/documents/{documentId}` → delete esignList record. The single-shot GET is allowed by DeepSign's rate limit (status endpoint is not rate-limited). |
| `withdrawn` / `rejected` | `DELETE /api/v1/documents/{documentId}` → delete esignList record. |
| `signed` | `DELETE /api/v1/documents/{documentId}` (best-effort; DeepSign auto-deletes after 14 d anyway) → delete esignList record and the archived `signedPdfPath` from Storage. The confirm dialog must explicitly warn that the signed PDF will be lost. |

All branches finish with a single Firestore transaction that deletes `esignList/{esignId}` and any associated Storage objects (`storagePath`, `signedPdfPath`). The deletion is logged to an audit collection `esignAudit/{tenantId}/deletions/{esignId}` with the deleting user, timestamp, prior status, and the optional `reason`.

### 4.5 `esignSendByEmail` (callable)

**Purpose:** Send the (signed) PDF via email to recipients chosen by the user. Used by the `document_send` actionsheet entry.

**Input:** `{ esignId: string; recipients: string[]; subject?: string; body?: string; includeSignedPdf: boolean }`

**Behaviour:** Refreshes the `documentUrl` via `GET /api/v1/documents/{documentId}`, downloads the (signed) PDF, dispatches the email via our existing mail-sending service with the PDF as attachment. Logs the dispatch in `esignList/{id}/sendLog`.

### 4.6 `esignWebhook` (HTTP `onRequest`)

**Purpose:** Receive DeepSign webhook callbacks for document- and signee-status changes.

**URL pattern:** `https://<region>-<project>.cloudfunctions.net/esignWebhook?esignId=<id>&token=<hmac>`

**Behaviour:**

1. Verify `token == HMAC-SHA256(esignId, WEBHOOK_SECRET)` — reject with 401 otherwise.
2. Parse the JSON body (DeepSign sends the relevant `documentId`, `documentStatus`, `signStatus`, and the changed signee).
3. Resolve the `esignList/{esignId}` doc; ignore if not found or already in a terminal state with the same status.
4. Update fields on the esignList record:
   - `documentStatus`, `signStatus`
   - For signee-status events: update the matching entry in `signees[]` (set `viewedTime`, `signedTime`, `completionTime`, `signStatus`, `signeeComment`)
   - Append an entry to `events[]` with `{ type, at, payload }` for the audit trail (see §6.1)
5. If terminal (`signed`, `rejected`, `withdrawn`), set `completionTime` and optionally trigger downstream automations (e.g. archive the signed PDF to long-term Storage — see §4.8).
6. Respond `200 OK` quickly. Errors should return `5xx` so DeepSign retries.

**Idempotency:** All updates are based on the latest event timestamp; duplicate webhook deliveries are no-ops.

### 4.7 `esignArchiveSigned` (Firestore trigger, `onUpdate`)

**Purpose:** When `documentStatus` transitions to `signed`, download the final signed PDF from DeepSign (`GET documentUrl`) and store it permanently in our Storage bucket at `tenants/{tenantId}/esign/{esignId}/signed.pdf`. Record `signedPdfPath` on the esignList document. DeepSign retains signed documents only 14 days — archiving is mandatory so that `document_view`, `document_share`, and `document_send` keep working past that window. No retention policy is enforced on our side; the signed PDF lives until the user invokes `esign_delete`.

## 5. Webhook Security

- The webhook URL contains an HMAC of the `esignId` (`token` query param). The CF rejects requests where the HMAC does not match.
- `WEBHOOK_SECRET` is stored in Secret Manager and rotated yearly.
- Optionally, DeepSign source IPs may be allow-listed via Cloud Armor when stable IPs become available.
- All webhook payloads are appended verbatim to `events[]` for traceability.

## 6. Firestore Data Model

### 6.1 `esignList/{esignId}`

```ts
interface EsignRecord {
  esignId: string;
  tenantId: string;
  ownerUserId: string;          // who initiated the signing
  source: 'user-upload' | 'cf-generated';
  sourceRef?: string;           // e.g. invoiceId, contractId
  documentName: string;
  storagePath: string;          // original PDF in Storage
  signedPdfPath?: string;       // after archive

  // DeepSign side
  deepsignDocumentId: string;
  documentStatus: 'uploading' | 'draft' | 'in-progress' | 'signed' | 'withdrawn' | 'rejected' | 'error';
  signStatus?: 'on-hold' | 'pending' | 'in-progress' | 'signed' | 'rejected';
  signatureMode: 'timestamp' | 'advanced' | 'qualified';
  jurisdiction: 'zertes' | 'eidas';
  comment?: string;
  initiatorAliasName: string;

  signees: Array<{
    signeeId: string;
    email: string;
    name?: string;
    signOrder: number;
    signStatus: 'on-hold' | 'pending' | 'in-progress' | 'signed' | 'rejected';
    viewedTime?: Timestamp;
    signedTime?: Timestamp;
    completionTime?: Timestamp;
    signeeComment?: string;
  }>;

  events: Array<{
    type: 'created' | 'started' | 'signee-viewed' | 'signee-signed' | 'signee-rejected' | 'completed' | 'withdrawn' | 'error';
    at: Timestamp;
    signeeId?: string;
    payload?: Record<string, unknown>;
  }>;

  createdAt: Timestamp;
  updatedAt: Timestamp;
  startedAt?: Timestamp;
  completionTime?: Timestamp;
  errorMessage?: string;
}
```

### 6.2 Security rules (sketch)

- Read: `request.auth.token.tenantId == resource.data.tenantId`. UI-level menu permissions gate which users can navigate to the page at all; Firestore enforces only the tenant boundary.
- Write: **denied for clients**; all writes happen via Cloud Functions only. The CFs additionally verify that the caller's permission profile grants the e-sign capability before performing destructive actions.

### 6.3 Indexes

- Composite: `(tenantId asc, createdAt desc)` — for the list page.
- Composite: `(tenantId asc, documentStatus asc, createdAt desc)` — for status filters.

## 7. Client UI

### 7.1 EsignListPage

**Route:** `/esign`

**Layout:** standard list page with a header (title + filter chips + "Send for signing" CTA in context menu) and a virtualized list of esignList records, ordered by `createdAt desc`.

**Per-row content:**

- **Left:** PDF icon, colored by status.
- **Title (line 1):** `documentName`.
- **Subtitle (line 2):** "Signed by N / M · last activity <relative time>" or "Waiting for <next signee email>".
- **Right:** status chip — one of `Draft`, `In progress`, `Signed`, `Withdrawn`, `Rejected`, `Error`. Tooltip shows `signStatus`.

**Context menu (top right of the page):**

- **Upload & send PDF for signing** — opens a file picker (PDF only, ≤ 40 MB), then a form sheet to capture `initiatorAliasName`, `comment`, `signatureMode`, `jurisdiction`, `sendMail`. On submit:
  1. Uploads PDF to Storage at `tenants/{tenantId}/esign/staging/{uuid}.pdf`.
  2. (Optional) Calls `esignScanPredefined` and shows a confirmation panel ("Detected N signees: …") with a "Send" button. If no signees are detected, blocks with an error message explaining the PDF needs Text Field Patterns.
  3. Calls `esignSendDocument`.
  4. On success, the new row appears at the top of the list via the live subscription.

**Empty state:** illustration + short copy + the same "Upload & send PDF for signing" button.

**Filtering:** status chips (`All`, `In progress`, `Signed`, `Rejected`, `Draft`, `Withdrawn`).

**Live data:** subscribe to `esignList` filtered by `tenantId` only. The page is **tenant-wide** — every user who can reach it sees every signing process in the tenant. Access is gated upstream via menu permissions; only privileged users see the menu entry.

### 7.2 Row click → Actionsheet

A click on a list entry opens an actionsheet with the following actions. Actions that aren't applicable to the current `documentStatus` are hidden (rules in parentheses).

| Action key | Label | Shown when | Behaviour |
|---|---|---|---|
| `esign_status_view` | View signing status | always | opens `EsignViewModal` |
| `document_view` | View PDF | always | opens PDF in viewer. Uses `previewUrl` (refreshed via `esignGetDocumentDetails`) while in progress, or `signedPdfPath` from our archive bucket once signed. |
| `document_share` | Share | `documentStatus = signed` | opens native share sheet with the signed PDF (from `signedPdfPath`). |
| `document_send` | Send via email | `documentStatus = signed` | opens compose dialog → calls `esignSendByEmail`. |
| `esign_delete` | Delete signing process | always (with status-dependent semantics, see §7.4) | confirm dialog → calls `esignDelete`. Removes the signing process from DeepSign **and** the esignList record. If the document is `in-progress`, the dialog explicitly warns the user that the process will be withdrawn first. |

There is **no edit action**. Once a PDF is uploaded to DeepSign the signing process is frozen by design — to correct the document, the user deletes the signing process via `esign_delete` and re-uploads the corrected PDF as a new signing process.

### 7.3 EsignViewModal

Full-screen modal opened by `esign_status_view`.

**Header:** document name, status chip, close button.

**Body — two columns on desktop, stacked on mobile:**

**Left column — Document preview**
Embedded PDF viewer using `previewUrl` from `esignGetDocumentDetails`. URL is short-lived (~1 min); the modal silently refreshes on tab focus.

**Right column — Signing process**
- **Summary block:** initiator name, signature mode, jurisdiction, created at, started at, completion time.
- **Signees timeline:** ordered list, each entry shows:
  - Name + email
  - Per-signee status chip (`On hold`, `Pending`, `In progress`, `Signed`, `Rejected`)
  - `Viewed: <date>` / `Signed: <date>` / `Rejected: <date>`
  - Optional signee comment
  - Avatar with status colour
- **Events timeline (collapsible):** rendered from `events[]`, useful for audit/support.

**Footer actions** (visibility depends on status):

- **Refresh** — re-fetch details (calls `esignGetDocumentDetails`).
- **Resend invitation to <signee>** — for each pending signee (calls a thin CF `esignResendInvitation` → `POST /signees/{signeeId}/resend-invitation`).
- **Download signed PDF** — when `documentStatus = signed`.
- **Delete signing process** (destructive, confirm dialog) — always available, mirrors the `esign_delete` actionsheet entry. Same status-dependent semantics as in §4.4.

### 7.4 Delete confirmation dialog

Triggered by `esign_delete` (list actionsheet) or the **Delete signing process** button in EsignViewModal. The dialog adapts to the current `documentStatus`:

| Status | Title | Body | Confirm label |
|---|---|---|---|
| `uploading` / `error` | Discard upload? | "This upload never reached DeepSign and will be removed locally." | Discard |
| `draft` | Delete signing process? | "The document will be deleted from DeepSign. No invitations have been sent yet." | Delete |
| `in-progress` | Withdraw and delete? | "The signing process will be withdrawn first, then deleted. Any signees who haven't signed yet will receive a withdrawal notification from DeepSign. This cannot be undone — to correct the document, upload the corrected PDF as a new signing process." | Withdraw & delete |
| `signed` | Delete signed document? | "**This will permanently delete the signed PDF.** Make sure you have downloaded a copy if you need one." | Delete signed PDF |
| `withdrawn` / `rejected` | Delete signing process? | "This will remove the signing process and its history." | Delete |

The confirm button is rendered in the destructive color. For the `signed` and `in-progress` variants, a "Download signed PDF" / "Download (unsigned) original" secondary button is offered alongside the destructive confirm so the user has a chance to keep a copy.

## 9. Edge Cases & Constraints

- **File size:** enforce ≤ 40 MB at upload-time client-side and CF-side (DeepSign hard limit).
- **MIME type:** only `application/pdf` is accepted for v1 (DOCX/XLSX out of scope).
- **Short-lived download URLs:** `documentUrl` is valid ≈ 1 min; always refetch via `GET /documents/{id}` before serving to the client.
- **DeepSign lifecycle:** draft = 7 d, unsigned in-progress = 30 d (auto-withdrawn), signed/withdrawn/rejected = 14 d. The `esignArchiveSigned` trigger ensures signed PDFs survive the 14 d deletion.
- **No predefined fields:** if `scan-predefined` returns an empty `signatureFields`, abort the upload and surface a clear UI error: "No signature fields found in this PDF."
- **Quota:** 10 000 documents / 24 h / company. Surface a friendly error if `429` is returned.
- **Rate limiting:** polling GETs throttled to once per 15 min — never poll in a loop; rely on webhooks. Use `GET /documents/{id}` only on explicit user action (modal open, refresh button).

## 11. Resolved Decisions

The following design decisions are locked in:

1. **Scope of EsignListPage:** tenant-wide. Every authenticated user who can reach the page sees every signing process for the tenant. Access is controlled upstream by menu permissions on privileged users; no per-user scoping or `esign.viewer` role is implemented in the page itself.
2. **Invitation email language:** rely on the DeepSign default behaviour (typically derived from the signee or initiator profile). No `language` field is set when creating signees, and no UI to choose it is offered.
3. **Retention:** no retention policy on the archived signed PDFs. Archived files in `tenants/{tenantId}/esign/{esignId}/signed.pdf` live until the user deletes the signing process via `esign_delete`. No long-term archive bucket, no scheduled cleanup.
4. **Edit-after-upload:** not supported. Once a PDF is uploaded to DeepSign the signing process is frozen. To correct a document, the user invokes `esign_delete` on the list entry (which withdraws first if the process is `in-progress`, then deletes from DeepSign) and re-uploads the corrected PDF as a new signing process. There is no `document_edit` action anywhere in the UI.
