# Resource Domain

## Overview

The Resource domain models physical objects owned by a subject (person, org, group). Examples include rowing boats, lockers, keys, cars, real estate, pets, and generic items. Resources can be reserved and are linked to ownerships. The model is intentionally generic: `type` and `subType` are category-driven so the system can be extended without schema changes.

## Firestore Collection

Collection name: `resources`

## Field Semantics

| Field | Type | Description |
|---|---|---|
| `bkey` | string | Firestore document ID (stripped on write, re-attached on read) |
| `tenants` | string[] | Multi-tenancy isolation; queries always filter by tenantId |
| `isArchived` | boolean | Soft-delete flag |
| `name` | string | Display name / identifier (e.g. locker number, key number, plate) |
| `index` | string | Computed search index string (generated by `getResourceIndex`) |
| `tags` | string | Comma-separated tag list for filtering |
| `description` | string | Detailed description |
| `type` | string | Resource type (from `resource_type` category: `boat`, `rboat`, `car`, `locker`, `key`, `realestate`, `pet`, etc.) |
| `subType` | string | Sub-type within the resource type (e.g. gender for lockers, `RowingBoatType` for rowing boats, car type for cars) |
| `usage` | string | Usage category (from `rboat_usage` category) |
| `currentValue` | number | Current monetary valuation |
| `load` | string | Max load specification |
| `weight` | number | Weight in kg |
| `color` | string | Hex color code |
| `brand` | string | Manufacturer brand |
| `model` | string | Model name/number |
| `id` | string | Serial number, VIN, plate number, etc. |
| `seats` | number | Number of seats |
| `length` | number | Length in cm |
| `width` | number | Width in cm |
| `height` | number | Height in cm |
| `data` | BaseProperty[]? | Extensible key-value metadata |

## Resource Types

The store pre-filters the collection by type into typed computed signals:

| Computed signal | `type` value | Notes |
|---|---|---|
| `boats` | `'boat'` | Generic boats |
| `rboats` | `'rboat'` | Rowing boats; `subType` is from `rboat_type` category |
| `cars` | `'car'` | Vehicles |
| `lockers` | `'locker'` | Storage lockers; `subType` is used for gender filter |
| `keys` | `'key'` | Physical keys |
| `realestate` | `'realestate'` | Properties |
| `pets` | `'pet'` | Animals |

## ResourceListStore

NgRx Signal Store (provided at component level). State:

| State field | Description |
|---|---|
| `searchTerm` | Text filter on `index` |
| `selectedTag` | Tag chip filter |
| `selectedResourceType` | Type filter (`'all'` or a specific type) |
| `selectedSubType` | Sub-type filter (e.g. rowing boat type) |
| `selectedGender` | Gender filter (for lockers) |

Key resource:
- `resourceResource` — real-time stream from `resources` collection ordered by `name`.

Computed per-type filtered lists (e.g. `filteredBoats`, `filteredRboats`, `filteredLockers`, etc.) apply text search, sub-type, gender, and tag filters.

Key actions:
- `add(isTypeEditable, readOnly)` / `edit(resource, isTypeEditable, readOnly)` — open `ResourceEditModalComponent`.
- `delete(resource, readOnly)` — call `ResourceService.delete`.
- `export(type)` — not yet implemented.

## ResourceEditStore

Separate NgRx Signal Store for the edit page (`resource-edit.page.ts`) used when editing a single resource directly on its own page (as opposed to in a modal).

## Components

| Component | Description |
|---|---|
| `ResourceListComponent` | Generic filterable list |
| `KeyListComponent` | Specialised list view for keys (`type = 'key'`) |
| `LockerListComponent` | Specialised list view for lockers (`type = 'locker'`) |
| `RowingBoatListComponent` | Specialised list view for rowing boats (`type = 'rboat'`) |
| `ResourceEditModalComponent` | Full-modal edit form (used by list) |
